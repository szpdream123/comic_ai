# Episode Asset Conversation History Design

> Date: 2026-05-31  
> Owner posture: INTJ / 长期主义 / Framework Architect  
> Status: Approved direction from product owner  
> Scope: 以资产为粒度持久化图片与视频生成对话历史，支持重新进入同一 tab 后自动回填完整线程

## 1. 背景与目标
当前 episode workbench 的资产生成结果主要停留在前端状态里，切换资产 tab 后只能看到当前内存中的结果，刷新后历史会丢失。用户希望把同一资产下的完整对话保留下来，并且明确区分：
- 用户发送内容靠右显示
- 系统返回内容靠左显示
- 用户 / 系统需要有清晰标识
- 用户发起的请求、生成中的任务卡、最终成功/失败结果都要完整保存
- 图片和视频都走同一套历史模型，避免后续再改表结构

本设计的目标是把“资产生成对话历史”升级为可持久化、可回放、可扩展的统一能力，使得：
- 再次点击某个资产 tab 时，自动按该 tab 对应的 `assetId` 和 `mediaMode` 拉取历史并渲染
- 图片和视频共用一套 schema、API 和 UI 渲染协议
- 在高并发写入下保持可查询、可分页、可扩展，而不是依赖全表扫描或单行热点更新

## 2. 设计原则
1. 统一模型优先。图片与视频不拆分表，不拆分消息协议。
2. append-only 优先。消息主体只追加，不做大段覆盖。
3. 线程与消息分离。线程表存定位信息，消息表存内容与状态变化。
4. 查询必须走索引。历史回填按线程主键定位，禁止全表扫描。
5. 预留扩展，不提前过度设计。先满足当前 workbench 场景，但 schema 要允许后续分区或归档。

## 3. 推荐方案

### 3.1 数据模型
采用两张核心表：

- `episode_asset_conversation_threads`
  - 一条记录对应一个资产线程
  - 唯一键：`organization_id + project_id + episode_id + asset_id + media_mode`
  - 记录线程元数据、创建人、更新时间、最新消息时间

- `episode_asset_conversation_messages`
  - 一条记录对应一条消息或一次状态变化
  - `message_type` 取值：
    - `user_request`
    - `task_status`
    - `result`
  - 保存消息正文、任务信息、结果状态、时间戳与创建人

### 3.2 消息语义
- `user_request`
  - 右侧用户请求卡
  - 记录 prompt、上下文、资产信息、生成参数快照
- `task_status`
  - 系统侧任务中卡
  - 记录 taskId、running/queued/succeeded/failed 等状态变化
- `result`
  - 系统侧最终结果卡
  - 记录成功或失败输出，成功时包含资源引用，失败时包含错误码与错误信息

### 3.3 API 形态
新增统一历史接口：
- `GET /api/creator/episode-asset-conversations?projectId=&episodeId=&assetId=&mediaMode=`
- `POST /api/creator/episode-asset-conversations`

写入由后端服务统一封装，前端只负责提交“请求开始”“任务状态更新”“结果完成”三类事件，不直接操作表结构。

## 4. 运行流程
### 4.1 首次生成
1. 用户在某个资产 tab 下提交图片或视频生成请求。
2. 后端创建或复用对应 thread。
3. 写入 `user_request`。
4. 写入 `task_status = running`。
5. 任务完成后写入 `result = success` 或 `result = failure`。

### 4.2 切换资产 tab
1. 用户点击另一个资产 tab。
2. 前端根据当前 `assetId` 与 `mediaMode` 请求历史。
3. 后端按 thread 唯一键取回消息列表。
4. 前端按时间顺序重建对话框，保持右侧用户卡、左侧系统卡的布局。

### 4.3 刷新与重进
刷新后不依赖 localStorage 恢复业务内容。所有对话内容都以后端历史为准，前端只保留 UI 级别的临时草稿状态。

## 5. 高并发与容量约束
用户担心 1 万到 10 万请求时表会不会扛不住，这里明确：**能扛，但前提是写法正确**。

### 5.1 为什么可行
- 写入模式是 append-only，消息行不会被频繁整行重写。
- 查询只按 thread 唯一键和消息索引取数，不做全表扫描。
- thread 数量增长的是资产线程数，不是单行热点。
- 10 万级请求的压力主要在索引与 IO，而不是“表名叫不叫一张表”。

### 5.2 需要避免的做法
- 不要把所有历史塞进一个 JSON 大字段里
- 不要靠一张表既存线程又存全部消息
- 不要用全局单行保存“当前最新状态”
- 不要把历史列表按无索引条件直接拉全量

### 5.3 预留扩展
如果后续消息量继续上升，可在不改 API 的前提下做：
- 按 `organization_id` 或 `created_at` 分区
- 热冷数据归档
- 最近 N 条快速查询，历史分页加载

## 6. 前端渲染约定
### 6.1 用户卡
- 靠右显示
- 标签为“用户”
- 展示 prompt 主体
- 展示必要的生成参数与上下文摘要

### 6.2 系统卡
- 靠左显示
- 标签为“系统”
- running 状态显示任务进行中卡
- 完成后显示结果卡
- 失败时保留失败结果，方便回看

### 6.3 资产切换体验
切换 asset tab 时，右侧对话区必须切换到该资产自己的历史，不允许串到上一个 tab 的内容。

## 7. 错误与一致性
- 重试提交同一动作时，后端必须保证消息不会重复插入成多条业务记录。
- 任务状态更新必须能幂等更新同一 task 的进度。
- 结果写入必须与任务完成态保持一致，失败结果不能被后续成功结果覆盖掉历史痕迹。
- 读取失败时前端应保留当前草稿，不清空输入框。

## 8. 测试策略
必须覆盖以下三层：
- schema 测试：确认线程表、消息表、唯一键与索引存在
- service / HTTP 测试：确认能写入、能查询、能按 assetId + mediaMode 回放
- web 渲染测试：确认资产切换后能加载正确历史，且用户/系统左右分布正确

## 9. 范围边界
本设计只解决资产生成对话历史，不负责：
- 导出历史
- 项目级全局聊天
- 其他业务模块的消息系统
- 复杂的实时协作编辑

## 10. 结论
采用“统一线程 + 两表消息模型 + mediaMode 区分”的方案，既能满足图片/视频共用历史，又能支撑高并发写入与后续扩展。  
这套方案不会把未来锁死在单一结构上，也不会因为先做了图片就迫使视频再改表。
