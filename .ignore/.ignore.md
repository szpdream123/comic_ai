# AI 漫剧创作平台与视频模型中转站架构记录

> 本文档记录架构讨论中的关键判断、约束和决策。目标不是一次性写完大而全方案，而是持续沉淀长期正确的系统边界。

## 设计原则

- 坚守第一性原理：先定义业务本质、用户、约束和不可妥协点，再选择技术。
- 使用马斯克五步工作法：
  1. 质疑需求；
  2. 删除不必要部分；
  3. 简化和优化；
  4. 加速周期；
  5. 自动化。
- 坚守长期主义：优先选择能支撑多租户、规模化、可观测、可计费、可扩展模型接入的架构边界。
- 不追求口头上的“100% 自信”。每个设计都需要列出假设、漏洞、验证方式和修复方案，直到剩余风险有明确边界。

## 当前已确认决策

### 产品方案约束

架构设计必须以现有产品方案为输入约束，不能只基于 AIComicBuilder 代码和平台架构推导。

当前已纳入参考：

- `docs/product/reelmate-core-replication-prd.md`
- `docs/product/reelmate-technical-integration-notes.md`

P0 产品闭环以 PRD 为准：

> 创建项目 → 上传/粘贴剧本 → 剧本解析 → 角色/场景/道具资产提取 → 资产确认与编辑 → 分镜拆解 → 风格校准 → 单个/批量生成分镜图 → 单分镜编辑与重生成 → 单分镜图转视频 → 任务状态展示 → 基础额度校验 → 导出素材包。

架构文档中任何与 PRD 冲突的 MVP 范围判断，都需要回到 PRD 修正。

### 业务边界

平台长期支撑两大业务：

1. AI 漫剧创作平台：面向内容生产者，从剧本、角色、分镜、关键帧、视频片段到成片交付。
2. 视频模型中转站：统一接入多个视频/图像/文本模型，提供路由、限流、计费、观测、故障转移和标准 API。

### 产品节奏

- 架构按“双业务底座”设计。
- 第一阶段产品露出优先做 AI 漫剧创作平台。
- 视频模型中转站第一阶段作为内部平台能力建设，但保留未来独立商业化能力。

### 上游 AIComicBuilder 定位

`twwch/AIComicBuilder` 适合作为创作流水线、UI、Prompt 和领域模型参考，但不适合作为长期平台底座直接扩展。

已观察到的上游特征：

- Next.js 单体应用；
- SQLite + Drizzle；
- 本地文件存储与 FFmpeg；
- SQLite 任务表轮询；
- 进程内 Provider 工厂直接调用模型；
- 模型配置和 API Key 更偏个人工具场景；
- 适合原型和单用户/轻量部署，不适合直接承载多租户、计费、SLA、统一模型网关和批量生产。

### 首批目标用户

优先面向内容工作室/MCN，兼容个人创作者。

该选择意味着以下能力必须成为一等公民：

- 多租户与团队权限；
- 多项目、多剧集、批量生产；
- 任务队列与失败恢复；
- 资产版本管理；
- 成本核算与额度控制；
- 生成质量评估；
- 模型路由和供应商故障转移；
- 操作审计与可观测性。

### 第一阶段交付承诺

第一阶段优先交付：

> 从剧本到高质量分镜图、单分镜视频片段与素材导出包。

具体包括：

- 剧本/大纲导入；
- 角色解析与角色设定；
- 场景和道具资产提取；
- 世界观、画风、色彩和镜头语言约束；
- 分集、场景、镜头拆解；
- 风格校准；
- 分镜提示词生成与可编辑；
- 分镜图生成；
- 单分镜图转视频；
- 资产版本管理与人工挑选；
- 素材包导出。

视频制作工具在早期较依赖 `AIComicBuilder` 的现有能力，后续逐步自主迭代。架构上需要把视频生成视为可替换模块，而不是把核心领域模型绑定死在上游实现上。

该选择的长期含义：

- 核心壁垒放在“创作资产可控性”和“生产工作流稳定性”，而不是单次视频模型调用。
- 视频生成可以先集成、后替换；资产、任务、Prompt、模型路由、计费和观测必须从第一天按平台能力设计。
- 第一阶段降低端到端成片失败风险，先验证内容团队是否愿意为高质量生产前置资产付费。

### AIComicBuilder 集成策略

采用“平台主干独立 + AIComicBuilder 作为视频制作适配器/子系统”的策略。

不直接 Fork 后在其代码上持续魔改，也不在第一阶段完全从零复刻其视频制作能力。

边界原则：

- 平台拥有自己的领域模型：租户、团队、项目、剧集、角色、场景、镜头、资产、任务、账单、模型调用记录。
- AIComicBuilder 只作为早期视频制作能力提供方，通过 Adapter 调用。
- 平台不依赖 AIComicBuilder 的 SQLite 表结构、前端状态、本地文件路径或任务轮询机制。
- 输入给 AIComicBuilder 的是平台定义的标准作业描述，输出回平台的是标准化资产和任务结果。
- 后续可将该 Adapter 替换为自研视频流水线、第三方模型网关，或多个制作后端并存。

长期目标：

- AIComicBuilder 从“依赖对象”逐步降级为“可替换 Provider”。
- 平台核心资产、工作流、计费、权限和观测不随视频制作后端变化而重写。

### 多租户与 B 端基础

第一阶段数据库和后端必须按 B 端多租户设计，UI 可以先简化为单团队体验。

原因：

- 首批目标用户是内容工作室/MCN，本质上是团队生产，不是个人玩具。
- 项目、资产、模型额度、账单、成员权限和审计都必须有明确归属。
- 如果先做完全单用户，后续迁移到 B 端会涉及数据隔离、权限、账单和审计的大面积重构。

初始边界：

- `Tenant / Organization`：商业客户或工作室。
- `Workspace / Team`：组织下的生产空间，可承载项目、成员和额度。
- `User`：自然人账户。
- `Membership`：用户在组织/团队内的角色。
- `Project`：漫剧项目，必须归属于租户/团队。
- `Asset`：角色图、参考图、首尾帧、视频片段、成片等，必须归属于租户/项目。
- `Job / Task`：所有 AI 生成与制作任务，必须归属于租户/项目，并可追踪发起人。
- `Usage / Billing Event`：所有模型调用、存储、转码和任务消耗都必须可归因到租户。

第一阶段 UI 策略：

- 可以默认每个新用户创建一个组织和一个工作区。
- 可以暂时隐藏复杂组织管理。
- 但 API、数据库、任务系统、资产系统和计费记录不能省略租户维度。

### 主数据存储

平台主库从第一天采用 PostgreSQL。

原因：

- B 端多租户、团队权限、任务状态、资产版本、计费事件和审计日志需要强一致关系模型。
- 模型调用、任务幂等、额度扣减和失败补偿需要事务能力。
- JSONB 可以承载模型参数、Prompt 变量、供应商响应等半结构化数据，同时保留关系查询能力。
- 后续报表、成本分析、项目检索、运营后台都更适合基于 PostgreSQL 演进。

边界：

- PostgreSQL 是平台唯一主数据事实来源。
- SQLite 不作为平台主库。
- 如果早期集成 AIComicBuilder，其内部 SQLite 只能是 Adapter 的执行细节，不能成为平台领域数据依赖。
- 平台需要定义自己的迁移体系、数据访问层和审计策略。

### 对象存储与全球化兼容

平台从第一天采用对象存储管理大资产。

资产包括：

- 剧本源文件；
- 角色参考图；
- 风格图、Mood Board；
- 分镜图；
- 首帧、尾帧；
- 视频片段；
- 音频、BGM；
- 成片与导出包；
- 供应商原始输出文件和调试文件。

原则：

- 数据库只保存资产元数据，不保存大文件本体。
- 资产元数据包括租户、项目、资产类型、版本、hash、尺寸、时长、MIME、存储 Provider、Region、Bucket、Object Key、访问策略和生命周期状态。
- 平台资产系统是唯一资产主权方。
- AIComicBuilder Adapter 和其他生成后端只能通过签名 URL 或受控上传/下载 API 访问资产。
- 不允许业务代码到处拼接云厂商 URL。

存储抽象：

- 定义统一 `ObjectStorageProvider` 接口。
- 国内可接入阿里云 OSS、腾讯云 COS、火山 TOS、七牛、UCloud 等。
- 海外可接入 AWS S3、Cloudflare R2、GCP Cloud Storage 等。
- 开发和测试环境可使用 MinIO 或本地 S3 兼容服务。

全球化预留：

- 租户需要有 `homeRegion` 或数据驻留策略。
- 资产写入时根据租户、业务区域和合规策略选择存储 Region。
- 访问时通过签名 URL、CDN 或代理层分发。
- 资产元数据中必须记录存储区域，避免未来跨境数据迁移不可追踪。
- 需要预留国内/海外模型供应商、支付、短信/邮件、CDN 和对象存储的可替换 Provider 边界。

长期目标：

- 支持国内客户数据留在国内。
- 支持海外客户数据留在海外。
- 支持未来按租户迁移、复制、归档或删除资产。
- 支持模型生成任务在靠近数据和模型供应商的区域执行，降低传输成本与延迟。

### 模型中转站阶段策略

第一阶段模型中转能力优先服务内部 AI 漫剧创作平台，不急于对外开放 API。

但架构和数据模型需要预留未来独立商业化能力。

第一阶段需要建设的内部模型网关能力：

- 统一模型调用接口；
- 文本、图像、视频模型的 Provider 抽象；
- 国内与海外供应商的可替换接入；
- 模型路由策略；
- 失败重试与故障转移；
- 请求幂等；
- 调用日志；
- 成本估算与实际成本记录；
- 租户/项目/任务级用量归因；
- 基础限流和额度控制；
- 供应商响应归档；
- Webhook/异步任务回调适配。

暂缓对外暴露的能力：

- 公共开发者门户；
- 外部 API Key 自助管理；
- OpenAI-compatible 公开 API；
- 开发者文档；
- 外部 SLA 承诺；
- 公开计费套餐；
- 外部风控和滥用检测面板。

预留的数据模型：

- `ApiCredential`；
- `ModelProvider`；
- `ModelEndpoint`；
- `ModelRoutePolicy`；
- `ModelInvocation`；
- `UsageEvent`；
- `RateLimitBucket`；
- `QuotaLedger`；
- `WebhookEndpoint`；
- `BillingAccount`。

长期策略：

- 先用自家漫剧生产业务打磨模型网关。
- 当内部稳定性、成本曲线、失败恢复和观测能力成熟后，再开放外部 API。
- 外部 API 不应绕过内部网关，而应复用同一套路由、计费、限流和观测底座。

### 任务系统

采用 PostgreSQL 负责“真相”，Redis/BullMQ 负责“调度”的任务系统。

职责边界：

- PostgreSQL 保存所有任务最终状态、业务归属、成本、审计、输入、输出、错误和重试记录。
- Redis/BullMQ 只作为高性能执行器，用于任务分发、延迟执行、重试、并发控制、优先级和 Worker 协调。
- 任何业务查询、审计、计费、恢复和用户可见状态都以 PostgreSQL 为准。
- Redis 中的状态不可作为长期事实来源。

核心表建议：

- `Job`：业务任务，如剧本解析、角色抽取、分镜生成、首尾帧生成、视频生成、转码、导出。
- `JobAttempt`：每次执行尝试，记录 Worker、开始/结束时间、错误、供应商请求 ID、耗时。
- `JobDependency`：任务依赖关系，支持 DAG。
- `ModelInvocation`：模型调用记录，可关联 JobAttempt。
- `UsageEvent`：计费和额度事件。
- `Asset`：任务输入输出资产。

关键设计要求：

- 所有任务必须具备租户、项目、发起人和幂等键。
- 创建任务时先写 PostgreSQL，再投递 Redis/BullMQ。
- Worker 执行前必须从 PostgreSQL 锁定或声明 JobAttempt。
- Worker 完成后先写入 PostgreSQL 状态、资产、成本和错误，再确认队列任务。
- 失败重试要区分模型供应商失败、业务参数失败、系统失败和用户取消。
- 支持人工重试、跳过、取消、恢复和重新生成。
- 支持优先级：交互式任务优先于批量任务，付费高等级租户可有更高并发。

长期演进：

- 第一阶段使用 BullMQ 满足 Node 技术栈下的队列调度。
- 当工作流 DAG、长周期异步任务、补偿逻辑和跨服务编排复杂度上升时，再评估 Temporal 等工作流引擎。
- 即使未来引入 Temporal，PostgreSQL 仍保留业务事实和审计数据。

### 服务形态

第一阶段采用模块化单体，运行时拆必要进程。

不在业务刚起步时过早微服务化。

运行时形态：

- `Web/API`：负责前端、BFF、认证会话、业务 API。
- `Worker`：负责异步任务执行、模型调用、资产处理、转码、导出。
- `AIComicBuilder Adapter`：负责隔离调用 AIComicBuilder 相关能力，可独立进程或独立容器。
- `PostgreSQL`：平台主库。
- `Redis`：BullMQ 调度与缓存。
- `Object Storage`：资产存储。

代码边界：

- `identity`：用户、组织、团队、成员、权限。
- `projects`：项目、剧集、场景、镜头、角色。
- `assets`：资产元数据、版本、存储、签名 URL。
- `jobs`：任务事实、状态机、依赖、尝试记录。
- `model-gateway`：模型 Provider、路由、调用、成本、限流。
- `billing`：额度、用量、账单事件。
- `production`：漫剧生产流程、Prompt、资产包、AIComicBuilder Adapter。
- `admin/ops`：运营后台、审计、观测。

原则：

- 模块之间通过明确的应用服务接口调用，不跨模块直接访问内部表和实现细节。
- 数据库可以物理同库，但表归属和访问边界需要清楚。
- Worker 复用同一领域服务，不另写一套业务逻辑。
- AIComicBuilder 不能反向依赖平台内部模块，只能通过 Adapter Contract 通信。

长期演进：

- 当模型网关外部商业化时，可优先拆出 `model-gateway`。
- 当资产访问规模变大时，可拆出 `assets` 服务。
- 当计费复杂度上升时，可拆出 `billing`。
- 拆服务的前提是模块边界已稳定，而不是为了架构形式感。

## 重要修正：MVP 必须围绕 AIComicBuilder 的核心能力

架构设计不能脱离 MVP 对 `AIComicBuilder` 当前能力的依赖。

`AIComicBuilder` 当前核心能力不是一个抽象的视频 Provider，而是一条已经成型的漫剧生产流水线：

```text
剧本输入
→ 剧本解析
→ 角色提取
→ 角色四视图
→ 智能分镜
→ 参考帧/首尾帧生成
→ 视频提示词生成
→ 逐镜头视频生成
→ 视频合成与字幕
```

MVP 应优先复用这些真实能力和控制点，而不是提前设计一个脱离现有实现的理想化创作流程。

修正后的原则：

- 第一阶段不擅自重写 AIComicBuilder 已验证过的生产链路。
- 平台主干负责 B 端地基：租户、项目、资产、任务、权限、用量、审计和对象存储。
- AIComicBuilder Adapter 负责承接其擅长的创作流水线能力。
- 我们的标准领域模型需要能映射 AIComicBuilder 的现有概念：Project、Episode、Character、Scene、Shot、Dialogue、ShotAsset、PromptTemplate、Agent、Task。
- MVP UI 和流程需要尽量贴近 AIComicBuilder 已有功能核心，再逐步增强 B 端能力。
- “底层结构化”不能先验决定；必须以 AIComicBuilder 当前可用流程和内容团队实际生产动作作为约束。

需要进一步确认的关键点：

- MVP 是直接复用 AIComicBuilder 的编辑器体验，还是只复用后端流水线？
- AIComicBuilder 是作为独立容器运行，还是以代码模块方式嵌入新平台仓库？
- 平台主库与 AIComicBuilder SQLite 之间如何同步或映射？
- 第一阶段哪些数据必须迁移到平台 PostgreSQL，哪些可以暂留 AIComicBuilder 内部？
- 哪些生产阶段必须纳入平台任务系统和成本记录，哪些可以暂时黑盒化？

### AIComicBuilder 复用路线

MVP 采用“偏 A 的 B”路线：

- 强复用 AIComicBuilder 的产品内核、编辑器体验和后端流水线；
- 平台外壳、B 端地基、租户、权限、项目入口、资产归档、任务观测、用量记录由我们建设；
- 不把 AIComicBuilder 永久视为核心内核；
- 长期必须逐步自研可替换的 Creative Production Kernel。

阶段策略：

1. `Wrapped Kernel`：AIComicBuilder 作为核心能力被平台包裹，解决 B 端入口、部署、配置、数据归属和最小观测。
2. `Controlled Kernel`：平台逐步接管资产、任务、模型调用、成本记录和关键状态机，AIComicBuilder 只负责执行部分生产步骤。
3. `Owned Kernel`：平台拥有完整创作生产内核，AIComicBuilder 降级为参考实现、迁移来源或备用 Provider。

关键原则：

- MVP 不脱离 AIComicBuilder 当前功能核心。
- 每次接入都要设计退出路径，避免把上游内部实现变成平台长期依赖。
- 能抽象输入输出 Contract 的地方先抽象，不能抽象的地方先隔离。
- 用户价值优先于架构洁癖，但不允许引入无法迁移的核心数据依赖。

### 仓库与集成形态

MVP 采用 Monorepo 双 App 形态：

- `platform app`：我们的 B 端平台主应用，负责租户、团队、项目入口、资产、任务、模型网关、用量、权限、运营后台。
- `aicomic kernel app`：基于 AIComicBuilder 的创作内核应用，负责复用其编辑器体验和生产流水线。
- `packages/*`：逐步沉淀共享 SDK、类型、UI、模型 Provider、Prompt 工具、资产 SDK、任务 SDK、Adapter Contract。

推荐结构示意：

```text
apps/
  platform/
  aicomic-kernel/
packages/
  contracts/
  config/
  db/
  storage/
  jobs/
  model-gateway/
  production-sdk/
  ui/
  observability/
```

原则：

- 不把 AIComicBuilder 直接混入平台 App 内部，避免长期耦合。
- 不把 AIComicBuilder 完全当外部黑盒，避免无法共享类型和逐步迁移。
- 平台与 kernel 通过清晰 Contract 通信。
- 共享能力从 `packages` 中长出来，而不是在两个 App 里复制粘贴。
- 后续自研 Creative Production Kernel 时，优先替换 `aicomic-kernel` 内部实现，而不重写平台主干。

长期目标：

- 从 AIComicBuilder 中逐步抽取可复用能力进入 `packages/production-sdk`、`packages/model-gateway`、`packages/jobs`、`packages/storage`。
- 形成自有核心工具箱。
- 最终平台可以接入多个 production kernel：AIComicBuilder kernel、自研 kernel、第三方视频制作后端。

### Platform 与 Kernel 通信

第一阶段采用“界面务实集成 + 数据 Contract API”的通信方式。

决策：

- UI 层可以通过反向代理、子应用、跳转或 iframe 等方式快速集成 `aicomic-kernel` 编辑器体验。
- 核心数据交换必须通过标准 Contract API。
- `platform app` 与 `aicomic-kernel app` 不共享业务表结构。
- 禁止两个 App 直接读写同一组领域表。

需要定义的 Contract：

- `ProductionProject`：平台项目映射到 kernel 项目的标准描述。
- `ProductionEpisode`：分集结构。
- `ProductionCharacter`：角色设定、视觉描述、参考图。
- `ProductionScene`：场景描述、风格、色彩、灯光。
- `ProductionShot`：镜头、台词、时长、构图、运镜。
- `ProductionAsset`：角色图、分镜图、首尾帧、视频片段等。
- `ProductionJob`：一次生产动作或批量生产动作。
- `AssetManifest`：kernel 输出资产清单。
- `ProductionStatus`：kernel 内部生产状态的标准化映射。

原则：

- Contract 先覆盖 MVP 必需字段，不追求一次性完整。
- Contract 版本化，避免未来 kernel 升级破坏平台。
- 平台生成全局 ID，kernel 可以保留自己的内部 ID，但必须保存平台映射 ID。
- Kernel 输出的资产必须回传到平台资产系统归档。
- Kernel 状态可以有内部细节，但用户可见状态由平台标准状态机表达。

严禁：

- 平台直接依赖 AIComicBuilder 的 SQLite 表结构。
- Kernel 直接写平台核心业务表。
- 通过本地文件路径作为长期资产引用。
- 把 API Key、模型配置、Prompt 版本只保存在 kernel 内部。

### 模型配置与密钥管理

第一阶段由平台接管模型配置和 API Key。

`aicomic-kernel` 不作为模型密钥和模型配置的长期事实来源。

原则：

- 租户级或工作区级配置模型供应商、模型、额度和路由策略。
- API Key、Secret Key 等敏感信息只保存在平台安全存储中。
- Kernel 不持久化用户真实 API Key。
- Kernel 发起模型调用时，应通过平台模型网关代调，或使用平台签发的短期受限调用凭证。
- 所有模型调用都必须记录到平台 `ModelInvocation`。
- 所有成本和用量都必须归因到租户、项目、任务和发起人。

安全要求：

- API Key 加密存储，不能明文落库。
- 日志、错误、调试信息中严禁输出完整密钥。
- 支持按租户禁用/轮换密钥。
- 支持供应商级别的限流、熔断、故障转移。
- 支持按模型、租户、项目做预算和额度控制。

对 AIComicBuilder 的改造要求：

- 移除或绕过其前端本地模型配置作为生产路径的依赖。
- Provider 工厂需要可被平台网关替代。
- 模型调用参数、Prompt、输入资产、输出资产、供应商响应 ID 都要能回传平台。
- 对于短期无法改造的调用点，需要通过 Adapter 包装，避免真实密钥暴露给 kernel 持久层。

模型调用接管深度：

- 采用平台模型网关代发模式。
- Kernel 所有文本、图像、视频模型请求都通过平台模型网关。
- Kernel 不直接持有供应商真实 API Key。
- Kernel 不直接调用供应商生产接口，除非是开发调试环境下显式绕过。

要求：

- AIComicBuilder Provider 工厂需要改造成 `PlatformGatewayProvider` 或等价 Adapter。
- Kernel 调用模型时传递平台短期会话、项目、任务、Prompt、模型能力类型和输入资产引用。
- 平台模型网关负责选择供应商、注入真实密钥、执行请求、记录调用、归档响应和返回标准结果。
- 所有模型调用必须生成 `ModelInvocation`。
- 所有可计费调用必须生成 `UsageEvent`。
- 视频异步模型必须由平台模型网关管理供应商 task id、轮询、Webhook 和最终归档。

长期意义：

- 模型中转站从第一天就是真实底座。
- 成本、限流、路由、故障转移和供应商观测不依赖 kernel 内部日志。
- 未来开放外部 API 时复用同一套模型网关。

### Prompt 资产化

Prompt 是平台的核心生产资产。

它不应只是代码里的字符串、用户浏览器本地配置，或 AIComicBuilder kernel 内部的附属数据。

第一阶段策略：

- 尊重 AIComicBuilder 现有 Prompt 编辑和模板能力，避免打断 MVP 生产链路。
- 平台需要同步、归档或接管 Prompt 的关键数据。
- Prompt 版本、使用记录、关联模型、输入上下文、输出结果和人工评价需要进入平台资产与观测体系。
- 后续逐步演进为平台统一 Prompt Registry，kernel 只消费平台解析后的 Prompt 和参数。

Prompt 需要具备的归属：

- 租户级 Prompt；
- 工作区级 Prompt；
- 项目级 Prompt；
- 剧集/场景/镜头级 Prompt Override；
- 系统内置 Prompt；
- Kernel/Provider 专用 Prompt。

Prompt 资产元数据：

- `promptKey`；
- `scope`；
- `version`；
- `content`；
- `variables/schema`；
- `targetStage`：如剧本解析、角色抽取、分镜、首帧、尾帧、视频提示词、质检；
- `modelFamily` / `providerCompatibility`；
- `createdBy` / `updatedBy`；
- `qualityNotes`；
- `usageStats`；
- `linkedInvocations`。

长期目标：

- 支持团队沉淀 Prompt 方法论。
- 支持 Prompt A/B 测试和效果评估。
- 支持按项目风格、题材、角色一致性自动选择 Prompt。
- 支持从成功项目中反向沉淀可复用 Prompt 模板。
- 支持未来自研 kernel 复用同一套 Prompt Registry。

### 角色一致性资产

角色一致性资产由平台接管，`aicomic-kernel` 负责生成、编辑和生产辅助能力。

原因：

- AI 漫剧质量高度依赖角色一致性。
- 角色是内容工作室/MCN 的长期 IP 资产，不应被锁在单个 kernel 或单个项目内部。
- 角色资产需要跨项目复用、版本化、审核、归档和持续优化。

平台需要管理：

- `CharacterAsset`：角色主资产；
- `CharacterVersion`：角色设定版本；
- `CharacterVisualProfile`：外貌、发型、服装、体型、年龄、气质、表演风格；
- `CharacterReferenceImage`：四视图、头像、半身、全身、表情、动作、风格图；
- `CharacterCostume`：服装套装和场景服装；
- `CharacterRelation`：角色关系；
- `CharacterUsage`：在哪些项目、剧集、镜头、资产中被使用；
- `CharacterQualityReview`：一致性评价、人工修正记录。

第一阶段策略：

- 平台数据模型支持跨项目角色资产。
- UI 可以先以项目内角色管理为主，降低复杂度。
- Kernel 生成的角色图必须回传平台资产系统。
- Kernel 内部角色 ID 必须映射平台角色 ID。
- 角色 Prompt、参考图、生成模型、版本和人工修改都需要可追溯。

长期目标：

- 建立租户级角色/IP 库。
- 支持角色跨项目复用。
- 支持角色一致性评分。
- 支持角色设定随项目世界观派生，但保留来源链路。
- 支持从高质量生成结果反向更新角色资产。

### Platform 与 Kernel 的数据同步策略

MVP 采用“平台保存关键镜像数据，kernel 保存执行细节”的策略。

该策略承认 AIComicBuilder 在 MVP 阶段仍承担生产内核职责，同时避免平台完全黑盒化。

事实源划分：

- 平台是 B 端事实源：租户、用户、团队、权限、项目入口、资产主权、模型配置、Prompt 资产、任务外部状态、用量、成本、审计。
- Kernel 是生产执行事实源：编辑器内部状态、临时草稿、细粒度生成过程、AIComicBuilder 当前尚未抽象出来的执行细节。
- 平台保存关键镜像：项目、剧集、角色、场景、镜头、核心资产、任务状态、生产阶段状态和输出 Manifest。

一致性原则：

- 不追求跨系统强一致，追求可恢复的最终一致。
- 所有跨系统写入必须有幂等键。
- 所有 kernel 输出必须通过事件或 Contract API 回传平台。
- 平台状态不能被 kernel 随意覆盖，只允许按状态机合法流转。
- 资产最终以平台对象存储和 `Asset` 元数据为准。
- 成本和用量最终以平台 `UsageEvent / ModelInvocation` 为准。

需要的同步机制：

- `ProductionKernelBinding`：记录平台项目与 kernel 项目的映射。
- `KernelSyncEvent`：记录 kernel 到平台的事件流，如项目创建、角色更新、镜头更新、资产生成、任务完成。
- `SyncCheckpoint`：记录每个绑定的同步游标和最近成功时间。
- `ReconciliationJob`：定时对账平台镜像与 kernel 状态，发现缺失资产、卡住任务、状态不一致。
- `AssetManifest`：kernel 输出资产清单，由平台归档到对象存储。

已知风险：

- 平台和 kernel 项目状态不一致；
- 重复事件导致重复资产或重复任务；
- kernel 内部删除/修改导致平台镜像过期；
- 同步失败导致用户在平台看到旧状态；
- 任务完成但资产未成功归档；
- 平台取消任务但 kernel 仍在执行；
- kernel 版本升级导致 Contract 字段变化；
- 用户在 kernel 编辑器内修改数据后未同步到平台。

修复方案：

- 以平台状态机限制用户可见状态。
- 每个同步事件带 `eventId`、`kernelProjectId`、`platformProjectId`、`entityType`、`entityId`、`version` 和 `occurredAt`。
- 平台按 `eventId` 去重，按实体版本处理乱序事件。
- 所有资产按 hash/object key 去重。
- 任务取消需要平台写取消意图，kernel Worker 执行阶段轮询或接收取消信号。
- 对账任务必须能重新拉取 kernel 当前状态并修复平台镜像。
- Contract 版本化，kernel 升级前跑兼容测试。
- 关键生产数据同步失败时在平台显示“同步异常”，允许人工重试。

结论：

该策略会引入一致性复杂度，但相比完全黑盒或完全重写 kernel，是 MVP 阶段更合适的折中。只要事实源划分、幂等、事件、对账和状态机从第一天设计进去，风险可控。

### 认证与权限模型

第一阶段采用组织、工作区、项目三级权限模型。

层级：

- `Organization`：商业客户、工作室、MCN 或企业主体。
- `Workspace`：组织下的生产空间，可按团队、业务线、内容账号或区域划分。
- `Project`：具体漫剧项目，归属于某个 Workspace。

权限继承：

- 资产、角色、镜头、任务、Prompt、模型调用记录默认继承项目权限。
- 第一阶段不做资产级、镜头级独立权限，避免 MVP 过重。
- 后续如有外包、审片、客户协作场景，再扩展细粒度权限。

基础角色：

- `Owner`：组织所有者，管理账单、成员、全局配置。
- `Admin`：组织/工作区管理员，管理成员、项目、模型配置。
- `Producer`：制片/项目负责人，可管理项目、任务、资产和导出。
- `Creator`：创作者，可编辑剧本、角色、分镜、Prompt 和生成任务。
- `Reviewer`：审核者，可查看、评论、批准或驳回资产。
- `Viewer`：只读成员。

关键要求：

- 所有 API 必须校验租户、工作区、项目归属。
- Kernel 入口必须使用平台签发的短期会话或 JWT，携带 `userId`、`organizationId`、`workspaceId`、`projectId`、`role`。
- Kernel 不自行决定 B 端权限，只执行平台授权后的操作。
- 所有关键操作写审计日志：创建项目、邀请成员、修改模型配置、发起生成、删除资产、导出成片、修改 Prompt。

长期演进：

- 支持项目级成员覆盖；
- 支持外部协作者；
- 支持客户审片链接；
- 支持资产级审批流；
- 支持按角色配置模型额度和导出权限。

### 计费与额度

第一阶段采用组织总额度 + 项目级预算的模型。

项目级预算由用户选填；未设置时只受组织总额度约束。

原则：

- 组织是商业付费主体。
- 组织拥有总额度、余额、套餐或后付费账期。
- 项目可以设置预算上限，用于控制单部剧或单个生产项目成本。
- 所有消耗必须可归因到组织、工作区、项目、任务、模型调用和发起人。
- 额度和成本使用不可变 Ledger/Event 记录，不直接依赖简单余额字段作为唯一事实。

需要记录的消耗：

- 文本模型调用；
- 图像模型调用；
- 视频模型调用；
- Agent/工作流调用；
- 视频转码与合成；
- 对象存储容量；
- CDN/下载流量；
- 导出包生成；
- 失败重试成本；
- 供应商侧已扣费但平台任务失败的异常成本。

核心表建议：

- `BillingAccount`：组织账务主体。
- `CreditLedger`：额度充值、扣减、退款、调整。
- `UsageEvent`：每次可计费消耗事件。
- `ProjectBudget`：项目预算，可选。
- `CostEstimate`：执行前预估成本。
- `ProviderCostRule`：供应商和模型成本规则。

执行策略：

- 任务执行前做预算检查和成本预估。
- 成本不确定的异步视频任务，先冻结预估额度，完成后按实际成本结算，多退少补或记录差额。
- 项目预算达到阈值时提示，达到硬上限时阻止新生成任务。
- 组织总额度不足时阻止高成本任务。
- 失败重试是否计费需要按失败原因区分：用户参数问题、供应商已扣费、平台系统错误、供应商错误。

长期演进：

- 支持后付费账单；
- 支持项目成本报表；
- 支持模型/供应商成本对比；
- 支持成员级消耗分析；
- 支持预算预警；
- 支持客户维度利润核算。

P0 与长期设计的差异：

- P0 按 PRD 只要求生成前预估额度校验和成功后消耗记录。
- P0 失败任务默认不扣除额度，记录 0 成本失败事件。
- 额度冻结、供应商实际成本回填、多退少补、供应商已扣费但平台失败的异常成本处理，可作为 P1/P2。
- 架构需预留冻结和实际成本回填，但 MVP 不强制一次做完。

### 生产状态机

第一阶段采用项目、分镜、任务分层状态机。

不使用单一项目总状态表达所有失败；P0 使用局部失败状态，避免一个分镜失败拖垮整个项目。

项目状态以 PRD P0 为准：

```text
project_draft
  → script_input
  → parsing
  → assets_pending
  → assets_reviewing
  → shots_ready
  → calibration_pending
  → calibration_passed
  → image_generating
  → image_partial_failed
  → exportable
```

分镜状态以 PRD P0 为准：

```text
draft
  → ready
  → image_generating
  → image_completed
  → video_ready
  → video_generating
  → video_completed

image_generating → image_failed
image_completed → image_stale → image_generating
video_completed → video_stale → video_generating
video_generating → video_failed
```

任务状态以 PRD P0 为准：

```text
queued → running → succeeded
queued → canceled
running → cancel_requested → canceled / succeeded / failed
queued → failed
running → failed
```

核心规则：

- 运行中输入不可随意变更，避免生成任务和输入版本错位。
- 失败任务必须保留错误类型、错误信息和重试入口。
- 批量任务父状态由子任务聚合得出。
- 批量分镜图生成必须支持部分成功。
- 单分镜重生成新增资产版本，不覆盖旧版本。
- 修改分镜字段后，已有图/视频需要标记为 `stale`。
- 导出前必须做资产完整性检查，缺失项必须列出。

P0 任务类型：

- `script_parse`
- `asset_extract`
- `asset_reference_generate`
- `style_calibration`
- `shot_split`
- `shot_image_generate`
- `shot_video_generate`
- `export_package`

第一批必须补的缺口：

1. 平台项目与 kernel 项目的绑定、短期会话和权限校验。
2. 公共资产确认流：角色、场景、道具三类资产，关键角色/主要场景阻塞校准通过。
3. 道具资产一等公民化：至少平台侧要有 `ProductionProp` 和镜像/metadata。
4. 风格校准：3 个代表分镜、校准图生成、通过/跳过、阻塞批量生成。
5. 平台任务状态机：项目/分镜/任务分层状态，局部失败和部分成功。
6. 平台额度校验：组织额度 + 可选项目预算，创建生成任务前预估。
7. 资产归档：kernel 本地输出通过 `AssetManifest` 进入平台对象存储。
8. 模型调用代管：平台模型网关或短期凭证，避免 kernel 持久化真实 API Key。

当前不能达到 100% 自信的点：

- AIComicBuilder 的“场景参考帧/参考图模式”能否直接覆盖 PRD 的“分镜图”概念，需要跑通一次实际链路验证。
- 风格校准是否能用现有参考帧/首尾帧生成能力低成本拼出来，需要看用户接受的校准结果形态。
- 道具资产是否必须在 P0 有完整 UI，还是可先作为场景/分镜 metadata 管理，需要产品取舍。
- 平台接管模型网关对 AIComicBuilder Provider 工厂的改造量，需要进一步读调用点并做技术 Spike。

## 重要待办

### 风格校准落地方案

状态：暂缓决策，重要待办。

PRD 将风格校准列为 P0 阻塞规则：

- 从已拆解分镜中选择 3 个代表性分镜；
- 生成 3 张校准图；
- 用户通过后才能批量生成分镜图；
- Producer/Admin 可跳过，但需要二次确认和审计。

当前暂不强行确定实现方案。

待验证问题：

- AIComicBuilder 现有首尾帧、参考图或场景参考帧能力，哪一种最适合承接“校准图”？
- 校准图应由平台单独管理，还是作为特殊 shot asset 管理？
- 校准通过后是否需要锁定视觉风格、Prompt 版本、模型参数和参考资产版本？
- 跳过校准对生成质量和用户预期的影响有多大？
- 是否需要在 P0 严格阻塞批量生成，还是允许灰度配置？

候选方案：

1. 平台新增风格校准模块，调用 kernel 单图生成能力；
2. 复用 kernel 分镜页生成 3 个代表分镜图，再由平台标记校准通过；
3. P0 弱化为视觉风格确认，P1 再补完整校准。

后续进入系统架构设计前，至少需要明确其数据模型预留和状态机挂点。

### 道具资产落地策略

状态：已确认方向。

PRD P0 包含角色、场景、道具三类公共资产。

AIComicBuilder 当前角色和场景模型相对明确，道具更多存在于 Prompt、分镜描述和场景细节中，尚不是一等公民数据模型。

决策：

- 平台侧将道具作为一等公民资产预留。
- P0 Contract 包含 `ProductionProp`。
- P0 可先弱化道具 UI 和生成能力，不强行要求完整道具参考图闭环。
- Kernel 内部可暂时把道具作为 shot/scene metadata 和 Prompt 上下文使用。

平台需要支持：

- 道具候选抽取；
- 道具名称、描述、关键性标记；
- 道具与场景/分镜的关联；
- 道具是否关键；
- 道具确认状态；
- 可选参考图；
- 后续道具版本和生成历史。

P0 阻塞规则：

- 关键角色和主要场景未确认时阻塞校准通过。
- 关键道具未确认时不阻塞校准通过，但在相关分镜生成前提示。

长期目标：

- 重要道具参考图和一致性管理；
- 道具跨项目复用；
- 道具与角色动作、镜头信息点绑定；
- 关键道具的一致性检查。

### 分镜图业务资产抽象

状态：已确认方向。

分镜图是平台业务资产，不是 AIComicBuilder 的某个字段。

第一性原理：

- “分镜图”的本质是用户可查看、可编辑、可重生成、可导出、可用于图转视频的单镜头视觉资产。
- AIComicBuilder 的 `first_frame`、`last_frame`、`reference`、`scene_ref` 只是当前 kernel 的实现产物。
- 平台不能把业务语义绑定到某个 kernel 内部字段。

决策：

- 平台定义统一资产类型：`storyboard_image`。
- P0 UI、状态机、导出和图转视频都围绕 `storyboard_image`。
- Kernel Adapter 负责把 AIComicBuilder 的实际产物映射为平台 `storyboard_image`。
- 具体映射可按项目、kernel 版本或生成模式配置。

候选映射：

- `storyboard_image ← first_frame`：实现最简单，适合首帧驱动的视频链路。
- `storyboard_image ← reference`：更贴近单张分镜图概念。
- `storyboard_image ← scene_ref`：适合场景参考模式，但可能缺少角色动作信息。
- `storyboard_image ← selected_asset`：由用户或系统在多个 kernel 产物中选择当前分镜图。

平台需要记录：

- `assetType = storyboard_image`；
- `sourceKernelAssetType`；
- `sourceKernelAssetId`；
- `shotId`；
- `version`；
- `isCurrent`；
- `promptVersionId`；
- `modelInvocationId`；
- `generationMode`；
- `metadata`。

原则：

- 重生成新增 `storyboard_image` 版本，不覆盖旧版本。
- 修改分镜描述、角色挂载、场景挂载或关键道具后，当前 `storyboard_image` 标记为 `stale`。
- 图转视频优先消费平台当前 `storyboard_image`，而不是直接找 kernel 内部 first frame。
- 导出素材包导出平台 `storyboard_image`，并可附带 kernel 原始资产。

## 待补关键输入：产品方案

当前讨论已经形成了一批架构与集成决策，但这些决策主要来自长期架构判断、AIComicBuilder 能力分析和 B 端平台经验。

重要风险：

- 尚未系统性产出产品方案；
- 尚未明确 MVP 的用户旅程；
- 尚未定义第一版页面、功能优先级和验收标准；
- 尚未把内容工作室/MCN 的实际生产流程映射到产品流程；
- 尚未确认哪些 AIComicBuilder 功能必须原样保留，哪些可以隐藏、重组或延后；
- 尚未定义 MVP 的商业闭环：谁使用、为什么付费、以什么结果验收。

修正原则：

- 后续架构细节不能继续脱离产品方案推进。
- 需要先补一版产品方案，再回头校验当前架构决策。
- 产品方案不要求一次性完整，但必须回答 MVP 第一版“用户如何完成一次有价值的生产任务”。

产品方案需要覆盖：

- 目标用户画像；
- 第一阶段核心场景；
- MVP 成功标准；
- 核心用户旅程；
- 页面与模块清单；
- AIComicBuilder 功能复用清单；
- 平台新增能力清单；
- 不做事项；
- 付费与额度体验；
- 运营后台最小需求；
- 产品验收标准。

当前处理方式：

- 保留已确认的长期地基决策；
- 暂停继续深入低层架构状态机之前，先补产品方案；
- 产品方案完成后，对已有架构做一次反向校验，删除不必要部分，简化过度设计。

## 产品方案优先级修正

当前讨论已经确定了若干长期架构地基，但继续深入技术细节前，必须先补齐 MVP 产品方案。

问题：

- 如果没有明确用户路径，技术模块会失去优先级。
- 如果没有明确 MVP 交付结果，任务系统、权限、计费、状态机都可能过度设计。
- 如果没有明确 AIComicBuilder 在用户体验中的角色，Kernel 集成深度无法判断。
- 如果没有明确验收标准，就无法判断架构是否足够，也无法接近“事实上的 100% 自信”。

修正决策：

- 暂停继续深入底层技术细节。
- 先定义 MVP 产品方案：目标用户、核心场景、用户路径、功能范围、非目标、验收标准。
- 技术架构必须服务产品闭环，而不是先验展开。

需要补齐的产品问题：

- 第一版用户打开平台后完成什么任务？
- 用户从剧本到资产包的关键路径是什么？
- 哪些 AIComicBuilder 功能第一版必须保留？
- 哪些 B 端能力第一版必须露出？
- 哪些能力只做底层预留但不做 UI？
- 成功的 MVP 如何衡量？

### 计费与额度

第一阶段采用组织总额度 + 项目级预算。

原则：

- 组织统一购买额度或套餐。
- 项目可以设置预算上限。
- 所有 AI 生成、模型调用、转码、导出和存储消耗都必须归因到组织和项目。
- 额度和成本记录必须可审计、可追溯、可对账。

核心模型：

- `BillingAccount`：组织级账务账户。
- `CreditLedger`：组织额度流水，不直接覆盖余额。
- `ProjectBudget`：项目预算、已用额度、预警阈值。
- `UsageEvent`：不可变用量事件。
- `ModelInvocation`：模型调用明细。
- `CostEstimate`：任务执行前的成本预估。
- `CostActual`：任务完成后的实际成本。

执行规则：

- 发起高成本任务前先做预算检查。
- 预算不足时阻止任务或要求管理员确认。
- 批量任务需要估算总成本，并支持分批执行。
- 失败任务是否计费取决于供应商实际扣费，必须记录供应商侧状态。
- 重试任务必须关联原任务，避免重复统计业务成本。
- 项目预算耗尽后，默认停止新增生成任务，但不影响查看和导出已有资产。

用户可见能力：

- 组织额度余额；
- 项目已用预算；
- 单次任务预估成本；
- 模型调用明细；
- 失败重试成本；
- 项目成本报表；
- 预算预警。

长期演进：

- 支持工作区级预算；
- 支持成员级消耗统计；
- 支持套餐、后付费、预付费；
- 支持外部 API 调用计费；
- 支持国内/海外不同币种和供应商成本模型。

### 模型中转站开放节奏

第一阶段先建设内部模型网关，服务 AI 漫剧创作平台；暂不开放外部 API 商业化。

但架构和数据模型需要预留未来对外 API 能力：

- API Key；
- Project/Workspace 级调用额度；
- Rate Limit；
- Usage Event；
- Billing Event；
- Model Invocation Log；
- Webhook；
- Provider Routing；
- Failover；
- 异步任务查询；
- 幂等请求；
- 客户侧错误码与供应商错误码映射。

节奏原则：

- 第一阶段不为外部开发者文档、SDK、开放平台后台投入过多资源。
- 内部创作平台必须通过统一模型网关调用模型，不能绕过网关直接调用供应商。
- 当内部网关在真实生产任务中验证稳定后，再产品化为视频模型中转站。

长期目标：

- 创作平台是模型网关的第一个高价值客户。
- 模型网关通过真实业务流量沉淀路由、成本、质量和稳定性数据。
- 未来对外开放 API 时，不需要重写模型调用、计费和观测底座。
